### Security Findings

### [MEDIUM] SEC-001: Bash Policy Bypass via Shell Quoting/Escaping

**Status:** ✅ Fixed (2026-04-24)

**ID:** SEC-001  
**Location:** `packages/commit/src/bash-policy.ts:17`  
**Category:** Security  
**OWASP:** A05:2025  
**CWE:** CWE-693  
**Effort:** easy

**Problem:**  
The shell tokenizer preserves literal quotes inside tokens. Commands like `"git" commit` or `g"it" commit` produce tokens where `"git"` does not exactly match the string `"git"`, causing the policy classifier to skip the subcommand check and return `"allow"`.

**Impact:**  
A compromised or prompt-injected AI agent can bypass the controlled-commit workflow (`/commit`) and execute direct `git commit` or `git push` commands.

**Remediation:**  
Normalize tokens by stripping surrounding quotes and unescaping shell escapes before comparison:

```typescript
function normalizeToken(token: string): string {
  if (
    (token.startsWith('"') && token.endsWith('"')) ||
    (token.startsWith("'") && token.endsWith("'"))
  ) {
    token = token.slice(1, -1)
  }
  return token.replace(/\\(.)/g, "$1")
}
```

### [LOW] SEC-002: Silent Error Handling Masking Permission Issues

**ID:** SEC-002  
**Location:** `packages/frontend-developer/src/tools/load-skill.ts:27`, `packages/python-developer/src/tools/load-skill.ts:28`  
**Category:** Security  
**OWASP:** A10:2025  
**CWE:** CWE-755  
**Effort:** trivial

**Problem:**  
Bare `catch` blocks swallow all filesystem errors (including `EACCES`). If a skill file exists but is unreadable due to permissions, the loader silently falls back to the next candidate path, potentially loading unintended content.

**Remediation:**  
Check the error code before falling back:

```typescript
} catch (err: any) {
  if (err.code !== "ENOENT") throw err
  /* continue to next candidate */
}
```

### Performance Findings

### [MEDIUM] PERF-001: Synchronous Blocking I/O in Plugin Initialization

**Status:** ✅ Fixed (2026-04-24)

**ID:** PERF-001  
**Location:** `packages/*/src/index.ts`, `packages/code-review/src/index.ts`  
**Category:** Performance  
**Effort:** medium

**Problem:**  
Plugin initialization uses synchronous `readFileSync` extensively, blocking the event loop. The code-review plugin eagerly loads all 12 agent prompts and 4 command templates at config time regardless of which are actually used.

**Impact:**  
Slower plugin initialization; unnecessary memory usage for unused prompts.

**Remediation:**  
Implement lazy loading for agent prompts and command templates. Convert file reads to async where possible.

---

### [MEDIUM] PERF-002: Redundant Filesystem Existence Checks

**Status:** ✅ Fixed (2026-04-24)

**ID:** PERF-002  
**Location:** `packages/commit/src/index.ts:16`, `packages/code-review/src/index.ts:14`, `packages/*/src/tools/load-skill.ts`  
**Category:** Performance  
**Effort:** trivial

**Problem:**  
Multiple locations use `existsSync` followed by `readFileSync`, creating unnecessary I/O overhead and TOCTOU windows.

**Remediation:**  
Remove `existsSync` calls and handle `ENOENT` directly in `readFileSync` try/catch blocks.

### [MEDIUM] ARCH-002: Hand-Maintained JS Mirror of TS Source

**Status:** ✅ Fixed (2026-04-24)

**ID:** ARCH-002  
**Location:** `src/index.js` vs `src/index.ts`  
**Category:** Architecture  
**Effort:** medium

**Problem:**  
`src/index.js` is manually maintained to mirror `src/index.ts`. The JS file already diverges: it does not check `typeof plugin[key] === "function"` when collecting hook keys. This means development behavior (via TS) differs from published runtime behavior (via JS).

### [MEDIUM] MAINT-004: Silent Error Swallowing in `loadSkillContent`

**Status:** ✅ Fixed (2026-04-24)

**ID:** MAINT-004  
**Location:** `packages/frontend-developer/src/tools/load-skill.ts:27`  
**Category:** Maintainability  
**Effort:** easy

**Problem:**  
All filesystem errors are silently discarded. Only after all candidates fail does a generic error throw.

**Remediation:**  
Preserve the last error and attach it as cause:

```typescript
let lastError: Error | undefined
for (const candidate of candidates) {
  try {
    return readFileSync(candidate, "utf8")
  } catch (error) {
    lastError = error as Error
  }
}
throw new Error(`${namespace} skill file not found for: ${name}`, { cause: lastError })
```

### Cross-Verification Composite Findings

**Composite-1 [MEDIUM]: Skill Loading Subsystem — Duplicated, Unabstracted, and Inconsistently Tested**  
The security finding (silent permission masking) exists in two near-identical files because there is no shared skill-loading abstraction. The frontend-developer variant has less test coverage, making undetected regressions more likely. **Recommendation:** Extract a shared `skill-loader` utility with proper error propagation and comprehensive tests.

**Composite-2 [MEDIUM]: Root Entrypoint — Type-Unsafe, Divergent, and Blocking**  
Unsafe `any[]` types, unsafe indexed assignment, and the hand-maintained JS mirror create a maintenance liability. Synchronous I/O during initialization means any error blocks the entire plugin load. **Recommendation:** Auto-generate the JS entrypoint at build time and add runtime type guards.

**Composite-3 [LOW]: Template/Asset Loading — Systemic TOCTOU Pattern**  
The `existsSync` + `readFileSync` anti-pattern appears across multiple packages. While the challenger dismissed these as non-exploitable (static assets, hardcoded paths), the pattern is still a code smell that should be eliminated. **Recommendation:** Replace with atomic reads that handle `ENOENT` gracefully.
