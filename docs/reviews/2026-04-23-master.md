## Maintainability Findings

### [LOW] MAINT-001: `any`-typed config in code-review tests disables compile-time validation for `mode`

**Status:** ✅ Fixed (2026-04-23)

**ID:** MAINT-001  
**Location:** `packages/code-review/tests/plugin.test.ts:41,47`  
**Category:** Maintainability  
**Effort:** easy

**Problem:**  
The newly added `expect(config.agent[name].mode).toBe("subagent")` operates on `const config: any = { agent: {} }`. TypeScript provides zero compile-time validation for the `mode` property.

**Impact:**  
Tests pass at runtime but offer no type-level regression protection. SDK contract drift would silently break consumers.

**Remediation:**  
Replace `any` with a typed fixture. Note: this is a test fixture, not production code, and runtime assertions still validate correctness.

```typescript
// Before
const config: any = { agent: {} }

// After
const config: { agent?: Record<string, { mode?: string }> } = { agent: {} }
```

---

### [LOW] MAINT-002: Type widening in python-developer test loses literal union safety

**Status:** ✅ Fixed (2026-04-23)

**ID:** MAINT-002  
**Location:** `packages/python-developer/tests/plugin.test.ts:11–13,20`  
**Category:** Maintainability  
**Effort:** easy

**Problem:**  
The test fixture declares `mode?: string` instead of the SDK's literal union `mode?: "subagent" | "primary" | "all"`.

**Impact:**  
Test fixture documents an overly permissive contract. However, the runtime assertion `expect(...).toBe("primary")` validates the exact literal.

**Remediation:**  
Import the SDK's `AgentConfig` type or use the literal union in the test fixture type.

```typescript
// Before
agent?: Record<string, { description?: string; prompt?: string; mode?: string }>

// After
agent?: Record<string, { description?: string; prompt?: string; mode?: "primary" | "subagent" | "all" }>
```

**ID:** DOC-001  
**Location:** `AGENTS.md` (no section covers agent `mode`)  
**Category:** Documentation  
**Effort:** easy

**Problem:**
